# PLUGIN_SPEC_TOC.md (WordPress プラグイン仕様分割ガイド｜AI 伴走開発向け)

本ドキュメントは、**1つの巨大な `SPEC.md` に集約せず、AI 伴走開発に最適化された粒度で仕様を分割するための「推奨セクション構成」**を定義します。  
ここで定義するファイルはあくまで「標準形」であり、各プラグインごとに取捨選択・分割統合してかまいません。

---

## 1. 推奨ファイル一覧 (ベター〜ベストプラクティス)

最低限おすすめするのは、**6〜8ファイル程度** の分割です。

1. `SPEC_OVERVIEW.md`  
2. `SPEC_ARCHITECTURE.md`  
3. `SPEC_UI_AND_FLOWS.md`  
4. `SPEC_API_AND_INTEGRATION.md`  
5. `SPEC_DATA_DICTIONARY.md`  
6. `SPEC_I18N_AND_A11Y.md`  
7. `SPEC_TEST_AND_CICD.md`  
8. `SPEC_AI_COLLAB.md` (AI 伴走開発ルール)

### 1.1. SPEC_OVERVIEW.md (プロジェクトの存在理由)

- **目的**: プラグインの「存在理由」「解決したいユーザー課題」「スコープ/非スコープ」を明文化する。
- **AI 的メリット**:
  - 「このプラグインは何のためのものか ?」を毎回説明せずに済む。
  - 仕様追加時に「スコープ外」を AI に判定させやすくなる。

### 1.2. SPEC_ARCHITECTURE.md (コード構造と責務)

- **目的**: `includes/`, `src/`, Gutenberg ブロック、REST クラスなどの責務分担を、**WP_PLUGIN_SPEC.md の§2〜§5と紐づけて** 整理する。
- **内容例**:
  - 主要クラス図・ディレクトリ構造
  - 「どのレイヤーでどの WordPress API を使うか」の方針
  - 共通 Composer ライブラリ (Appendix: 「共通ライブラリを Composer 化」) の利用方針
- **AI 的メリット**:
  - 「この処理はどの層に置くべき ?」といった相談を AI に投げたとき、迷いなく提案してもらえる。

### 1.3. SPEC_UI_AND_FLOWS.md (管理画面・フロント UI と画面遷移)

- **目的**: `## 7. 管理画面の仕様` / `## 9. Gutenberg ブロック仕様` / `## 10. Classic エディター対応` を **プラグイン固有の画面設計に落とし込む**。
- **内容例**:
  - 管理画面のワイヤーフレーム／コンポーネント一覧
  - 各画面の入力項目・バリデーション・エラーメッセージ
  - ブロックの属性一覧と編集 UI の挙動
  - ユーザーフロー (例: 設定〜保存〜プレビュー〜公開)
- **AI 的メリット**:
  - React/TypeScript コンポーネントを生成するときに、**画面単位の仕様をそのままプロンプト化しやすい**。

### 1.4. SPEC_API_AND_INTEGRATION.md (REST API および外部サービス連携)

- **目的**: `## 8. REST API 仕様` をプラグイン固有のエンドポイント設計に具体化する。
- **内容例**:
  - `/plugin-slug/v1/...` ごとの
    - メソッド (GET/POST/PUT/DELETE)
    - パラメータ (型・必須／任意)
    - レスポンス JSON スキーマ
    - 認可ポリシー・`permission_callback` 条件
  - 外部 API (例: OpenAI, DeepL 等) の呼び出し仕様
- **AI 的メリット**:
  - 「このエンドポイントの PHP 実装を作って」「このレスポンスを使うフロントのフックを書いて」といったタスクを、**1エンドポイント単位で安全に委譲できる**。

### 1.5. SPEC_DATA_DICTIONARY.md (データ、設定値、ストレージ定義)

- **目的**: Options API、Post Meta、Custom Table など、**どこに何を保存するか** をカラムレベルで定義する。
- **内容例**:
  - `option_name` / `meta_key` / テーブルカラム名
  - 型、nullable、デフォルト値
  - どの画面や API から読み書きされるか
- **AI 的メリット**:
  - データ移行やリファクタリング時に、AI に **スキーマを誤解させない**。
  - TypeScript の型定義と1対1で対応させやすい。

### 1.6. SPEC_I18N_AND_A11Y.md (国際化・アクセシビリティ)

- **目的**: `## 4. 国際化` と `## 5.3. アクセシビリティ` を、プラグイン単位でどう守るかを明記する。
- **内容例**:
  - Text Domain、翻訳対象となる文字列の方針
  - RTL 対応の方針
  - キーボード操作や ARIA 属性の最低ライン
- **AI 的メリット**:
  - 文言生成や UI 実装を AI に任せたときに、「i18n/A11y の抜け」をレビューしやすくなる。

### 1.7. SPEC_TEST_AND_CICD.md (テスト戦略、CI/CD)

- **目的**: `COMMON_SPEC_CICD.md` を **WordPress プラグイン向けに翻案** した、自プロジェクト専用のテスト戦略を定義する。
- **内容例**:
  - PHPUnit / Playwright / Jest 等の採用方針
  - GitHub Actions のワークフロー構成 (lint / test / build / deploy)
  - 「どのブランチで何が走るか」
- **AI 的メリット**:
  - ワークフロー YAML の部分修正を AI に任せる際に、**守るべき全体方針** を明示できる。

### 1.8. SPEC_AI_COLLAB.md (AI 伴走開発ルール)

- **目的**: `WP_PLUGIN_SPEC.md` の Appendix A (Spec Driven Developing Rule) を、各プラグイン単位で具体化する。
- **内容例**:
  - 「AI に渡すときに必ず参照させるファイル一覧」
  - 「このプロジェクトで使う AI (Cursor / ChatGPT / Claude 等) の役割分担」
  - 標準プロンプトのテンプレート

## 2. 粒度の目安 (どこまで分割するか)

- **ベター・プラクティス**:  
  - 上記8ファイルのうち、**最低4〜5個** (Overview / Architecture / UI / API / Data) は分割する。
- **ベスト・プラクティス (AI 伴走を本格運用)**:
  - 8ファイルすべてを用意し、各タスクで「参照する仕様ファイル」を AI に明示してから依頼する。
- **やりすぎ例 (避けたい)**:
  - 1コンポーネントごとや1エンドポイントごとに個別ファイルを乱立させる。  
    → 「検索性が下がり、AI に読み込ませるファイル選定も複雑になる」ため非推奨。

---

## 3. AI へのプロンプト例

実装タスクを AI に依頼する際は、次のように **参照すべき仕様を事前にセットで指定** するのがおすすめです。

```text
以下の仕様に従って実装してください。

- WP_PLUGIN_SPEC.md
- docs/SPEC.md
- docs/SPEC_ARCHITECTURE.md
- docs/SPEC_UI_AND_FLOWS.md の「3.2 設定画面：スコア算出ルール編集」
- docs/SPEC_DATA_DICTIONARY.md の「4.1 options スキーマ」
- docs/SPEC_AI_COLLAB.md の「2. プロンプト運用ルール」
```

このように、「**共通仕様 (WP_PLUGIN_SPEC) + プラグイン固有仕様群**」を AI に束ねて渡すことで、**誤読を減らしつつ、分割された仕様の恩恵 (見通し・保守性) を保つ**ことができます。
